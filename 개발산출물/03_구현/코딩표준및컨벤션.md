## 1. 백엔드 (Python/Django)

| 항목 | 규칙 |
|---|---|
| 파일 인코딩 | UTF-8, 필요 시 `# -*- coding: utf-8 -*-` 명시(Windows 환경 콘솔 인코딩 이슈 방지) |
| 모델 필드명 | snake_case, `verbose_name`에 한글 라벨 + 단위 명시(예: `revenue = models.DecimalField('매출 (억원)', ...)`) |
| DB 테이블명 | `Meta.db_table`로 명시적 지정, 앱 접두어 유지 |
| 시계열 모델 규칙 | 월별은 `fiscal_year`+`fiscal_month`(+`unique_together`), 일별은 단일 date 필드 — `timeseries_augmentation.py`가 이 규칙에 의존하므로 신규 시계열 모델 추가 시 반드시 준수 |
| 관리 명령(Management Command) | `앱/management/commands/`에 위치, docstring에 사용법 명시, `--dry-run` 옵션 제공 권장 |
| 시크릿/자격증명 | 코드에 하드코딩 금지 — 운영 DB 비밀번호 등은 `settings.DATABASES`에서 재사용하거나 환경변수로 주입 |
| 예외 처리 | 서비스 함수는 `try/except` + `logger.error(..., exc_info=True)` 후 표준 오류 응답(`{'error': ...}, status=500`) 반환 |
| 주석 | 자명한 코드에는 주석을 달지 않는다. "왜"가 비직관적인 경우(숨은 제약, 특정 버그 우회, 놀랄 만한 동작)에만 짧게 남긴다 |

## 2. 프론트엔드 (React/TypeScript)

| 항목 | 규칙 |
|---|---|
| 컴포넌트 | 함수형 컴포넌트 + Hooks(`useState`/`useEffect`), `React.FC` 타입 명시 |
| 서비스 계층 | `src/services/*.ts`에 API 호출 함수 집중, 컴포넌트에서 직접 `fetch` 호출 금지 |
| **API 응답 계약** | `api.ts`의 `ApiService.get/post/put()`은 **파싱된 JSON을 그대로 반환**한다(axios의 `{data: T}` 래핑 아님). 신규 서비스 작성 시 `response.data`가 아니라 `response`를 직접 사용해야 한다 — 이 규칙을 어겨 실제 프로덕션 급 버그(`kpiService.ts` 전체 API가 항상 undefined 반환)가 발생한 이력이 있다 |
| 상태-API 응답 형태 일치 | 백엔드가 "단일 집계 객체"를 반환하는 API는 프론트에서도 단일 객체 상태로 받아야 한다(배열로 오인해 응답을 버리는 실수 이력 있음 — `Dashboard.tsx` 참조) |
| 방어적 렌더링 | 목업/폴백 데이터가 실제 타입과 다를 수 있으므로, 중첩 프로퍼티 접근에는 옵셔널 체이닝(`?.`)을 사용한다 |
| 라우팅 | 신규 메뉴 추가 시 `App.tsx`의 `menuItems`에 고유 `id`를 추가하고, 그 `id`가 곧 URL 경로(`/{id}`)가 된다 |
| 스타일 | Tailwind CSS 유틸리티 클래스 사용, 커스텀 CSS 최소화 |

## 3. Git / 커밋

| 항목 | 규칙 |
|---|---|
| 커밋 메시지 | 무엇을 했는지보다 **왜** 했는지 중심으로 1~2문장 요약 + 상세 본문 |
| 커밋 단위 | 기능/버그 단위로 분리, `--amend`는 아직 푸시되지 않은 직전 커밋의 오탈자 수정 등 예외적인 경우에만 사용 |
| 브랜치 | `main` 단일 브랜치 운영(현재 프로젝트 규모 기준) |
| 민감정보 | `.pem`, `venv/`, `db.sqlite3`는 `.gitignore`에 등록, 실수로 스테이징된 경우 즉시 unstage |

## 4. 데이터 증강/ETL 관련 특수 규칙

- 시계열 증강 엔진은 **YH 원격 DB에 절대 쓰기를 시도하지 않는다** — 읽기 전용 제약을 코드 레벨에서도 지킨다.
- 증강된 값은 과거 실측 패턴에서 크게 벗어나지 않아야 한다(추세 포화, 최솟값 하한 등 안전장치 필수).
- 실제 원본 데이터(YH 백업 등)를 다룰 때는 원본 테이블에 쓰기 작업을 하지 않고, 항상 로컬/운영 자체 DB로만 ETL한다.
