## 3. 계층 설계 원칙

### 3.1 도메인 앱 분리
각 업무 영역(생산/품질/영업/구매/재무/원가/ESG 등)을 독립된 Django 앱으로 분리하여 모델·시리얼라이저·뷰·URL을 자체 관리한다. 앱 간 참조는 최소화하고, 통합이 필요한 경우(대시보드 등) `erp_sync` 계층에서 여러 앱의 모델을 조회해 집계한다.

### 3.2 ERP 연동 계층 (erp_sync)
외부 YH ERP는 두 가지 방식으로 연동한다.

1. **정식 매핑 계층**: `ERPSource → ERPTableDefinition → ERPFieldDefinition → ERPTableMapping → ERPFieldMapping → ERPTargetModel/ERPTargetField` — 소스 테이블/컬럼과 내부 Django 모델/필드 간 매핑을 DB 레코드로 관리한다. 실제 복원된 YH 백업(MSSQL)에서 라이브로 조회한 컬럼 정의와, 실제 Django 모델을 introspect한 타겟 정의를 연결한다(`rebuild_mapping_master_data` 관리 명령).
2. **연결 제어**: `ERPConnectionConfigModel`이 각 소스(YH 원격, LOCAL_BACKUP)의 `is_enabled`/`use_fallback`/`fallback_source`를 관리한다. YH가 비활성화되어 있으면 자동으로 `fallback_source`(로컬 백업 DB)로 전환된다.

### 3.3 시계열 자동 증강 엔진
`utils/timeseries_augmentation.py`는 모델을 introspect하여(fiscal_year+fiscal_month 조합 또는 단일 date 필드) 대상 테이블을 자동 탐지하고, 각 그룹(차원 조합)의 과거 이력에서 추세(선형회귀)+노이즈를 계산해 오늘 날짜(+버퍼)까지 신규 행을 생성한다. 장기 추정이 발산하지 않도록 추세 반영을 일정 스텝에서 포화시키고, 금액/수량류는 과거 최솟값의 절반을 하한으로 둔다. Celery Beat가 매일 1회 실행한다.

### 3.4 프론트엔드 라우팅
`App.tsx`의 `getMenuIdFromLocation()`이 `window.location.pathname`에서 메뉴 ID를 읽어 초기 상태를 결정하고, `setActiveMenu` 래퍼가 상태 변경 시 `history.pushState`로 주소창을 동기화한다. `popstate` 리스너로 뒤로가기/앞으로가기를 지원한다. 기존 ~150개 메뉴 렌더링 분기(`if (activeMenu === 'x') return <X/>`)는 변경 없이 그대로 재사용된다.

## 4. 배포 아키텍처

```
AWS Lightsail (3.38.107.23, ax.claros.co.kr)
└── /home/ubuntu/netplus-mis (git 저장소)
    └── docker-compose.lightsail.yml
        ├── claros-backend   (Django, Gunicorn 2 workers, 메모리 절약)
        ├── claros-frontend  (Nginx, React 빌드 산출물 서빙 + SSL)
        └── netplus-postgres (pgvector/pgvector:pg15, 명명된 볼륨 보존)
```

- 프론트엔드 컨테이너의 Nginx가 SSL 종료 + `try_files ... /index.html` SPA 폴백을 수행한다.
- 백엔드/프론트엔드 컨테이너는 외부 `netplus-network`를 공유하며, DB 컨테이너는 배포 시 재사용(볼륨 보존)한다.
