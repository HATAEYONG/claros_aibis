## 2. 배포 아키텍처

```
/home/ubuntu/netplus-mis  (git 저장소, origin main과 동기화)
└── docker-compose.lightsail.yml
    ├── claros-backend   (Django + Gunicorn, GUNICORN_WORKERS=2)
    ├── claros-frontend   (Nginx: React 빌드물 서빙 + SSL 종료 + /api 리버스프록시)
    └── netplus-postgres  (pgvector/pgvector:pg15, 명명된 볼륨으로 데이터 보존)
```

- `backend`/`frontend` 컨테이너는 `netplus-mis_netplus-network`(외부 네트워크)를 공유한다.
- DB 컨테이너는 배포마다 재생성하지 않고 **볼륨을 보존**한다(운영 데이터 유실 방지).

## 3. 배포 전략

| 항목 | 방침 |
|---|---|
| 배포 방식 | `git pull` → 이미지 재빌드(`docker compose build backend/frontend`) → 컨테이너 재기동(`up -d`) |
| 무중단 여부 | 완전 무중단은 아님(컨테이너 재기동 중 수초~수십초 다운타임 발생) — 별도 블루/그린 구성 없음 |
| DB 마이그레이션 | 배포 후 `docker compose exec backend python manage.py migrate` 별도 실행 |
| 롤백 | 이전 이미지 태그 유지 시 `docker compose up -d`로 즉시 롤백 가능(현재는 `latest` 태그만 유지 — 개선 필요) |
| 시크릿 관리 | DB 비밀번호 등은 `docker-compose.lightsail.yml`의 `environment`에 명시(운영 서버 로컬 파일, git 미포함) |

## 4. 사전 점검 체크리스트

- [ ] 로컬에서 `python manage.py check` 통과
- [ ] 로컬에서 `npx tsc --noEmit` 통과(신규/변경 파일 기준)
- [ ] 로컬 브라우저 프리뷰로 핵심 화면 크래시 없음 확인
- [ ] 변경사항이 원격 YH DB에 쓰기를 시도하지 않는지 재확인
- [ ] git 커밋에 시크릿(비밀번호/키)이 포함되지 않았는지 확인
- [ ] 운영 DB 백업(`backup_prod_db.sh`) 완료 — 특히 대량 데이터 교체가 있는 배포의 경우 필수

## 5. 배포 승인 절차

운영 반영은 **PO(발주자) 명시적 승인** 후에만 진행한다. 특히 다음 행위는 반드시 별도 확인을 받는다.

- 운영 DB 테이블 전체 삭제(mass delete) 후 재적재
- git `origin/main`으로의 push
- 운영 컨테이너 재빌드/재기동
