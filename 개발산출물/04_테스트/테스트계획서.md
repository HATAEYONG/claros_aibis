# 테스트계획서

**작성일**: 2026-07-08 | **버전**: 1.0

---

## 1. 테스트 목적

- 백엔드 API가 요구사항정의서의 각 기능 요구사항을 충족하는지 검증한다.
- 프론트엔드 화면이 백엔드 응답을 정확히 소비·표시하는지(런타임 크래시 없이) 검증한다.
- 외부 ERP(YH) 연결 불가 상황에서도 전체 시스템이 정상 응답하는지(폴백 동작) 검증한다.

## 2. 테스트 범위

| 범위 포함 | 범위 제외 |
|---|---|
| 대시보드 5종 API, KPI 관리 API, ERP 연결설정/매핑 API | 부하테스트/성능테스트(별도 계획 필요) |
| 시계열 증강 엔진의 데이터 정합성 | 브라우저 호환성 매트릭스(Chrome 기준만 검증) |
| 프론트엔드 핵심 화면(대시보드/KPI관리) 렌더링 | 전체 147개 메뉴 화면 전수 UI 테스트 |
| 로컬→운영 배포 후 실제 공개 URL 검증 | 모바일 실기기 테스트 |

## 3. 테스트 환경

| 구분 | 환경 |
|---|---|
| 로컬 | Windows 10, Python venv, `python manage.py runserver`(8000), `npm run dev`(Vite, 3000), SQLite |
| 운영 | AWS Lightsail(3.38.107.23), Docker Compose, PostgreSQL, Nginx+SSL, `https://ax.claros.co.kr` |
| 브라우저 검증 도구 | Claude Preview(Chrome 기반 headless 제어) — console/network/snapshot/screenshot 확인 |

## 4. 테스트 유형 및 방법

| 유형 | 방법 | 도구 |
|---|---|---|
| 단위/기능 테스트 | Django `APIRequestFactory` + `force_authenticate`로 뷰 함수를 직접 호출해 상태코드/응답 필드 검증 | `manage.py shell -c` 스크립트 |
| 시스템 점검 | `python manage.py check` | Django 자체 |
| 정적 타입 검사 | `npx tsc --noEmit` | TypeScript |
| 통합/E2E | 실제 개발서버 기동 후 브라우저에서 페이지 로드·네트워크·콘솔에러 확인 | Claude Preview |
| 배포 후 검증 | 공개 URL에 대해 `curl`로 실제 응답 상태/본문 확인 | curl |
| 회귀 테스트 | 새 기능 반영 후 이전에 정상이던 화면/API 재확인 | 위 도구 조합 |

## 5. 완료 기준(Exit Criteria)

- 요구사항추적표에 매핑된 모든 REQ에 대해 최소 1개 이상의 테스트케이스가 통과해야 한다.
- 대시보드 5종 API가 오늘 날짜 기준 200을 반환하고, 서로 다른 두 날짜에 대해 값이 달라야 한다(정적 응답이 아님을 증명).
- 프론트엔드 핵심 화면(통합대시보드, KPI관리)이 콘솔 에러/ErrorBoundary 크래시 없이 렌더링되어야 한다.
- 운영 배포 시 컨테이너가 재시작 루프 없이 안정적으로 기동해야 한다.

## 6. 결함 심각도 기준

| 등급 | 기준 |
|---|---|
| 치명(Critical) | 화면 전체 크래시, API 500, 원격 DB 쓰기 시도(정책 위반) |
| 주요(Major) | 특정 화면/카드의 데이터가 항상 0 또는 하드코딩값 표시 |
| 경미(Minor) | 표시 단위/라벨 불일치, 콘솔 경고 |
