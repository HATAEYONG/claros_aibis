## 1. 장애 유형별 대응

### 1.1 사이트 접속 불가 (`https://ax.claros.co.kr` 무응답)

```bash
ssh -i <key.pem> ubuntu@3.38.107.23
docker ps    # 컨테이너가 떠 있는지 확인
```

| 상황 | 조치 |
|---|---|
| 컨테이너가 아예 없음/Exited | `docker compose -f docker-compose.lightsail.yml up -d` |
| 컨테이너가 재시작을 반복(Restarting) | `docker logs claros-backend --tail 100`으로 원인 확인 → 아래 1.2 참조 |
| Nginx(frontend) 컨테이너만 문제 | SSL 인증서 만료 여부 확인(`docker logs claros-frontend`) |

### 1.2 백엔드 컨테이너 재시작 반복(OOM 의심)

**과거 실제 발생 사례**: Gunicorn worker 5개가 각각 22개 AI 에이전트 프레임워크를 전체 로드하면서 1.9GB 메모리를 초과.

```bash
# 1) 메모리 사용량 확인
docker stats --no-stream claros-backend

# 2) worker 수 축소 (docker-compose.lightsail.yml에서 이미 GUNICORN_WORKERS=2로 설정되어 있어야 함)
grep GUNICORN_WORKERS docker-compose.lightsail.yml

# 3) 그래도 부족하면 스왑 확인/추가 (운영 서버 스왑 파일 설정 여부 확인)
free -h
```

### 1.3 대시보드가 0/빈 값만 표시

원인 후보 순서대로 확인:

1. **데이터 자체가 비어있는지**: `docker compose exec backend python manage.py shell -c "from reports.models import ExecutiveSummary; print(ExecutiveSummary.objects.count())"`
2. **시계열이 오늘 날짜까지 연장되지 않았는지**: `extend_timeseries` 수동 실행
3. **프론트엔드가 응답을 실제로 렌더링에 반영하는지**: 브라우저 개발자도구 Network 탭에서 API 응답 본문 확인 → 값이 있는데 화면에 0으로 보이면 프론트엔드 상태 매핑 버그(과거 `Dashboard.tsx`가 응답을 버리던 사례 참조, 03_구현/프로그램목록.md)

### 1.4 특정 화면 전체 크래시(흰 화면 또는 ErrorBoundary 메시지)

```
1. 브라우저 콘솔에서 정확한 에러 메시지와 파일:라인 확인
2. "Cannot read properties of undefined" 계열이면 대부분:
   - API 응답 형태와 프론트 기대 형태 불일치(response.data 오접근 등)
   - 인증 실패 시 폴백(목업) 데이터가 실제 타입과 다름
3. 해당 서비스 파일(src/services/*.ts)에서 api.get()의 반환값을
   그대로 쓰는지(.data 접근 없이) 확인
4. 수정 후 Vite 개발서버를 완전히 재시작(HMR 캐시가 stale한 에러를
   계속 보여줄 수 있으므로 preview_stop → preview_start로 프로세스
   자체를 새로 띄우고 확인)
```

### 1.5 KPI/대시보드 API가 500 반환

```bash
docker compose exec backend python manage.py shell -c "
from erp_sync.services.dashboard_data_service import DashboardDataService
from rest_framework.test import APIRequestFactory, force_authenticate
from django.contrib.auth import get_user_model
factory = APIRequestFactory()
user = get_user_model().objects.first()
req = factory.get('/x/', {'date': '2026-07-08'})
force_authenticate(req, user=user)
resp = DashboardDataService.get_sales_dashboard(req)
print(resp.status_code, resp.data)
"
```
트레이스백을 확인해 어느 모델/필드 접근에서 실패했는지 특정한다.

### 1.6 YH 원격 DB 관련 오류 로그가 보임

**원칙적으로 발생해서는 안 된다** — `dashboard_data_service.py`는 YH 원격 연결을 시도하지 않도록 재작성되어 있다. 만약 로그에 YH 연결 시도 흔적이 보이면:

1. `ERPConnectionConfigModel.objects.get(source_code='YH').is_enabled`가 `False`인지 확인
2. `False`인데도 시도한다면 최근 배포에서 `erp_connection_config.py`/`dashboard_data_service.py`가 되돌려졌는지(git diff) 확인
