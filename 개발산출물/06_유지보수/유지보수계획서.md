# 유지보수계획서

**작성일**: 2026-07-08 | **버전**: 1.0

---

## 1. 유지보수 범위

| 구분 | 내용 |
|---|---|
| 정기 유지보수 | 시계열 데이터 일일 자동 연장(Celery Beat), 로그/디스크 모니터링 |
| 결함 수정 | 사용자/모니터링을 통해 발견된 버그 수정 및 재배포 |
| 기능 개선 | 요구사항 추가/변경에 따른 화면·API 확장 |
| 데이터 정합성 관리 | ERP 매핑 메타데이터, 증강 시계열 데이터의 지속적 검증 |

## 2. 정기 점검 항목

| 주기 | 항목 | 방법 |
|---|---|---|
| 일 1회 | 시계열 데이터가 오늘 날짜까지 연장되었는지 | `extend_timeseries` 관리 명령(Celery Beat 자동 실행, 미가동 시 수동) |
| 주 1회 | 컨테이너 상태/메모리 사용량 | `docker ps`, `docker stats` |
| 주 1회 | 운영 DB 백업 | `deploy/backup_prod_db.sh` |
| 월 1회 | 대시보드 5종 API 응답 정합성(날짜별 값 변화) | curl 스모크 테스트 |
| 필요 시 | ERP 매핑 마스터 데이터 유효성 | `rebuild_mapping_master_data`(MSSQL 원본 접근 가능할 때) |

## 3. Celery Beat 미가동 시 대체 절차

운영 서버 메모리 제약(1.9GB)으로 Celery worker/beat가 상시 구동되지 않는 경우, 다음 중 하나로 대체한다.

1. **수동 실행**: 관리자가 주기적으로 `docker compose exec backend python manage.py extend_timeseries` 실행
2. **OS cron 등록**(권장 후속 조치): Lightsail 서버에 cron job으로 위 명령을 매일 새벽 1시 실행하도록 등록

## 4. 장애 등급 및 대응 시간(SLA 목표)

| 등급 | 정의 | 목표 대응시간 |
|---|---|---|
| 1등급(치명) | 전체 서비스 접속 불가, 데이터 유실 위험 | 즉시 대응, 1시간 내 임시조치 |
| 2등급(주요) | 특정 화면/API 오류로 핵심 업무 불가 | 당일 내 조치 |
| 3등급(경미) | 표시 오류, 성능 저하 | 다음 정기 배포에 반영 |

## 5. 변경관리 절차

1. 요구사항/결함 접수
2. 영향도 분석(요구사항추적표 참조하여 관련 설계/구현/테스트 범위 파악)
3. 로컬 개발·테스트(04_테스트 절차 준용)
4. PO 승인
5. 배포절차서에 따른 운영 반영
6. 사후 모니터링

## 6. 알려진 기술부채(우선순위 순)

| 항목 | 설명 | 권장 조치 |
|---|---|---|
| `masterDataService.ts` API 계약 불일치 | `kpiService.ts`와 동일한 패턴의 잠재적 버그 44곳 | 동일 방식으로 수정(별도 작업 등록됨) |
| Celery 상시 구동 미구성 | 운영 메모리 제약으로 보류 | 서버 메모리 증설 또는 별도 경량 워커 분리 검토 |
| 배포 이미지 태그 미관리 | `latest` 태그만 사용, 롤백 시 이전 이미지 없음 | 커밋 해시 기반 이미지 태깅 도입 |
| 손익(순이익) 완전 실측 아님 | 법인세 등 영업외 손익은 영업이익의 78% 근사치 | 실제 세무/비영업손익 데이터 확보 후 정교화 |
| 재고 단가 데이터 일부 근사 | 최근 구매단가가 없는 품목은 기본단가(1,000원) 대체 | 품목마스터 원가 데이터 확보 시 개선 |
