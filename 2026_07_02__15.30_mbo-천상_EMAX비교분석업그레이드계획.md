## 최상위 목표
"현재 DeepSeeHub(Claros MIS-AI) 플랫폼의 AI/Text-to-SQL 아키텍처를 SAP PRD 수준(LangGraph 멀티에이전트·SQL Guard·실제 LLM 기반 SQL 생성·멀티RDB)으로 끌어올려, 플랫폼 고도화와 SAP 같은 MSSQL 기반 고객사 대응력을 동시에 확보한다."

## 배경 / 참고자료 요약
- `SAP_MSSQL_Select_AI_Agent_Enterprise_PRD_v1.0-1.docx`: 1페이지 요약 — 프로젝트 목표(자연어→SQL, WorkSet SQL 재사용, 10M 분석, 멀티RDB), 아키텍처(Client→API→Agent→Ontology→KG→SQL Intelligence→MSSQL), 핵심 Agent 9종(Supervisor/Intent/Schema/WorkSet/SQL/Safety/Interpreter/10M/Report), 보안(SELECT 전용/DDL·DML차단/AuditLog/RBAC), 기술스택(Python 3.13·FastAPI·SQLAlchemy·LangGraph·Redis·PostgreSQL·MSSQL)
- `SAP_Enterprise_PRD_Document_Set-1.zip` (11개 문서): **실제로는 본문 없는 템플릿** — 모든 문서가 "목적/주요구성/요구사항/아키텍처/모듈설계/확장성/개발규칙/보안/테스트/로드맵"이라는 동일한 헤더만 반복하고 세부 스펙은 비어 있음. → 이번 계획은 요약 docx 1장의 컨셉만 신뢰할 수 있는 근거로 사용하고, 나머지는 우리 쪽에서 구체화해야 함.

## 현재 vs 목표 (AS-IS → TO-BE)

| 항목 | 현재 (AS-IS) | 목표 (TO-BE, 이번 1단계 범위) |
|------|-------------|-------------|
| Text-to-SQL | `backend/ai/chat_services.py`의 `_generate_sql()`이 키워드 매칭으로 4종 하드코딩 SQL 템플릿 반환 (LLM 미사용) | Claude(Anthropic) 기반, 스키마 인지 동적 SQL 생성으로 교체. `llm_service.py`의 기존 `generate_sql()` 경로를 실사용 경로로 통합 |
| SQL 안전장치 | 없음 — 생성된 SQL을 `connection.cursor().execute()`로 그대로 실행 (DDL/DML 포함 전부 허용) | 신규 SQL Guard 모듈: SELECT 전용 강제, DDL/DML/위험 키워드 차단, 화이트리스트 기반 파서 검증 |
| 오케스트레이션 | `agent_orchestration_service.py`의 커스텀 순차 실행기 (그래프 아님, 정규식 기반 라우팅) | LangGraph 기반 최소 파이프라인: Intent → Schema → SQL → Safety(Guard) → Executor → Interpreter 5노드 그래프 |
| 멀티RDB 연동 | `erp_sync/utils/erp_db_connector.py`에 PostgreSQL/SQLServer/MySQL/Oracle/SQLite 커넥터 클래스는 구현되어 있으나, AI 파이프라인과 미연동 (AI는 Django 기본 DB만 사용) | AI 파이프라인이 `ERPDatabaseConnector`를 경유하도록 통합 — 소스 타입 무관 동일 경로로 SQL Guard·실행·로그 적용. MSSQL은 코드 완성도 확보(실접속 검증은 후속) |
| 감사/RBAC | `security.AuditLog`, `governance.PolicyRule` 등 모델은 존재하나 AI-SQL 실행 경로와 미연동 | AI-SQL 실행마다 `AuditLog` 레코드 생성 (요청자·질의·생성SQL·차단여부·실행결과) |
| 스키마 레지스트리 | 없음 — 필요시 즉석 introspection | 범위 밖 (Phase 2로 이월, 로드맵에 명시) |
| WorkSet SQL 재사용 | 없음 | 범위 밖 (Phase 2로 이월) |
| 10M 분석 | 6M 온톨로지(Django ORM 관계형)만 존재, "10M" 카테고리 정의 없음 | 범위 밖 — PRD 자체에 10M 정의가 없어 PO 확정 필요 (Phase 3 후보로 로드맵에 기재) |

### 계획 대비 중요한 차이 (정직하게 기록)

1. **당초 계획한 `chat_services.py::text_to_sql`이 실제로는 urls.py에 연결되지 않은 죽은 코드였다.**
   발견 즉시 실제 도달 가능한 `ai/views.py::text_to_sql`로 작업 대상을 옮겨 목표의 실효성을 확보했다.
   (자기수정 게이트: "Task 실패"가 아니라 "Task 대상이 틀림"에 해당 — 같은 Task 번호 안에서 대상만
   교정, 전체 재실행 없이 해결)
2. **AS-IS 진단이 부분적으로 부정확했다.** "생성된 SQL이 안전장치 없이 그대로 실행됨"이라던 원래
   우려는 그 경로가 죽은 코드였으므로 실제 위험도는 낮았다. 대신 "라이브 경로 대부분이 목업"이라는
   더 근본적인 문제가 있었다. 로드맵 문서에 이 정정 내용을 명시했다.
3. **SAP ERP 동기화 코드가 이미 존재한다는 사실을 새로 발견**했다 (`erp_sync/services.py`).
   MBO 승인 시점에는 몰랐던 내용 — 로드맵 문서 2.2절에 기록.

### 미달성 항목

| 항목 | 사유 | 후속 조치 |
|------|------|----------|
| LLM 응답 품질 실검증 | `ANTHROPIC_API_KEY` 미설정 | API 키 확보 후 실제 자연어 질의 5건 이상으로 검증 필요 |
| MSSQL 실접속 검증 | 자격증명 없음 (PO 확인, 합의된 보류) | 실제 MSSQL 인스턴스/자격증명 확보 시 별도 세션에서 검증 |
| `query_database` 목업 툴 교체 | 이번 승인 범위 밖 (Phase 2로 명시) | 로드맵 문서 5절 1번 항목 — 실사용자 체감 임팩트가 가장 큰 다음 작업으로 제안 |

### 총평
SQL Guard·LLM 기반 Text-to-SQL·LangGraph 파이프라인·멀티RDB 통합·AuditLog 연동을 계획대로 구현했고,
작업 도중 "계획한 대상이 죽은 코드였다"는 사실을 발견해 실제 라이브 엔드포인트로 스스로 교정함으로써
목표의 실효성을 지켰다. LLM 실응답 품질과 MSSQL 실접속은 자격증명 부재로 이번 범위에서 검증하지
못했으며, 이는 미달성으로 정직하게 기록하고 로드맵에 후속 조치를 남겼다. 가장 중요한 다음 단계는
기본 라이브 채팅이 실제로 호출하는 `query_database` 목업 툴을 이번에 만든 파이프라인으로 교체하는
것이다.
